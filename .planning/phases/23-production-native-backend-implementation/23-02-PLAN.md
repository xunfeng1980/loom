<objective>
Move the Phase 20 textual `loom.decode` surface toward compiled ODS/TableGen
evidence while preserving default MLIR-free workspace builds.
</objective>

<context>
Phase 20 emitted deterministic textual `loom.decode` operations but did not
register a compiled dialect. MLIR's recommended operation path is ODS/TableGen.
Phase 23 should add reviewed ODS sources and strict-toolchain validation without
forcing every developer to build a C++ dialect library by default.
</context>

<threat_model>
- Threat: textual and ODS dialect surfaces diverge.
  Mitigation: add a manifest/test that maps Phase 20 textual op names and
  attributes to ODS records.
- Threat: compiled dialect work makes default builds require LLVM.
  Mitigation: keep ODS files as sources plus optional strict checks.
- Threat: ODS verifier semantics overclaim safety.
  Mitigation: ODS verifies operation shape only; artifact verifier and runtime
  policy remain the trust boundary.
</threat_model>
