<objective>
Define the production native backend request/report contract around Phase 22
runtime plans and cache keys before adding compiled dialect or JIT behavior.
</objective>

<context>
Phase 22 made `RuntimePlan` and `RuntimeCacheKey` mandatory backend inputs.
Phase 23 must not accept ad hoc lowering requests, DuckDB-specific state, or a
new implicit ABI. The first step is a host-neutral backend model in
`loom-native-melior`.
</context>

<threat_model>
- Threat: the backend bypasses runtime planning and compiles unsupported inputs.
  Mitigation: backend request construction requires a native-candidate runtime
  plan and cache key.
- Threat: backend identity is too weak for cache/fallback decisions.
  Mitigation: include backend version, LLVM/MLIR version, target/layout, pipeline
  ID, and capabilities in reports.
- Threat: C++/Rust convenience wrappers become the stable ABI.
  Mitigation: document wrappers as internal and keep `loom_runtime.h` unfrozen.
</threat_model>
