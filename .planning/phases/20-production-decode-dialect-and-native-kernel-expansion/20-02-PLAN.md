<objective>
Define the `loom.decode` dialect contract and emit deterministic dialect-shaped textual artifacts for the first production-supported primitive slice.
</objective>

<context>
The research recommends defining the dialect semantics before committing to a compiled C++/ODS dialect. Phase 20 should make the production surface visible and testable while keeping default builds independent of MLIR/LLVM.
</context>

<threat_model>
- Threat: textual dialect output is mistaken for registered MLIR dialect support.
  Mitigation: docs and reports must distinguish dialect contract/textual surface from optional compiled C++/ODS registration.
- Threat: dialect ops lose the verifier trust link.
  Mitigation: every emitted module carries artifact/facts identifiers, row bounds, feature set, and discharge status metadata.
- Threat: emission becomes non-deterministic.
  Mitigation: golden tests compare exact text for representative supported slices.
</threat_model>
