<objective>
Add coverage for bitpack, frame-of-reference, and numeric compression shapes
without claiming production native support prematurely.
</objective>

<context>
Loom has mature synthetic L1 `BitPack` and `FrameOfReference` decode semantics,
and Phase 20 defines native kernel names for bitpack/FOR but intentionally
rejects them as unsupported. Phase 21 must bridge real Vortex facts to Loom
coverage and identify what Phase 23 needs before native expansion is credible.
</context>

<threat_model>
- Threat: bitpack/FOR facts are incomplete but emitted as executable Loom
  artifacts.
  Mitigation: require bit width, logical count, offset/block facts, validity
  facts, and oracle equivalence before emission.
- Threat: Phase 20 native lowering starts accepting bitpack/FOR without a real
  native delta.
  Mitigation: keep production lowering tests asserting unsupported unless this
  phase deliberately adds and verifies the delta.
- Threat: numeric compression support gets conflated with Vortex's full ALP or
  PCodec semantics.
  Mitigation: record ALP/PCodec as facts/deferred unless Loom-owned params can
  be derived safely and tested.
</threat_model>
