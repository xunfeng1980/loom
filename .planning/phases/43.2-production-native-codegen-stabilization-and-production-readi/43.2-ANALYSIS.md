# Phase 43.2 Analysis: Why Production Stabilization Must Precede ABI Freeze

## Summary

Phase 43.1 should be treated as "real native codegen exists and has a focused
positive gate." Phase 43.2 should be treated as "the production system can rely
on this path without freezing the wrong ABI."

The distinction matters because Phase 44 is a contract freeze. Once Phase 44
freezes symbols, fields, ownership rules, diagnostics, and cache identity, later
discoveries become compatibility problems. Phase 43.2 is the last cheap place to
learn those lessons.

## What Is Solid After Phase 43.1

1. **No zero-buffer native positive path.**
   The focused 43.1 gate checks that the production Arrow semantic codegen path
   does not use `reference_zeroed_value_bytes` as positive evidence.

2. **Real melior ExecutionEngine path exists.**
   `execute_arrow_semantic_codegen_jit` lowers Arrow semantic buffer slots and
   calls a real JIT entry, `loom_arrow_semantic_codegen_buffers`.

3. **Phase 40 admission exists.**
   JIT-produced buffers are rebuilt into Arrow arrays and admitted only after
   native/model validation succeeds.

4. **Cache identity is no longer artifact-only.**
   Runtime/cache identity includes artifact, verifier, backend, pipeline, shape,
   and trace fingerprints.

These are necessary and meaningful. They are not the same as production
stability.

## What Is Still Too Thin For ABI Freeze

### 1. Helper-Path Evidence vs Production-Route Evidence

The positive 43.1 test proves the native JIT function and validation machinery
work. Phase 43.2 should prove the production route chooses, runs, validates,
caches, cancels, falls back, and diagnoses the real native path without relying
on helper-only assumptions.

Question for Phase 43.2:

- Can a supported artifact go through the same route a host would use, produce
  JIT buffers, validate them, and expose only stable positive evidence?
- Can unsupported, cancelled, corrupted, or divergent paths avoid partial
  positive route evidence?

### 2. Determinism Is Not Yet Load-Bearing

Phase 43.1 records cache identity, but Phase 43.2 needs repeated-run evidence:
same inputs should produce stable keys and trace/buffer digests; intentional
drift should produce a visible miss or fail-closed result.

Question for Phase 43.2:

- Are toolchain, pipeline, target, schema, projection, predicate, split, trace,
  and output digests all represented in the right place?
- Is there any drift that would accidentally hit cache?

### 3. Native Output Validation Needs Attacks, Not Just Agreement

Validation is meaningful only if it catches adversarial buffer defects. Phase
43.1 has positive and some divergent-output coverage. Phase 43.2 should expand
negative injection at the Arrow buffer boundary.

Question for Phase 43.2:

- Does validation catch wrong buffer length, wrong bitmap length, wrong null
  count, schema drift, offset drift, trace drift, and bridge/container
  malformed cases?
- Are zero-row, one-row, and multi-column nullable cases stable?

### 4. Resource Semantics Are ABI Inputs

Buffer ownership, release lifetime, cancellation behavior, concurrency,
alignment, and cache replay are not performance polish. They are ABI inputs.
Phase 44 cannot freeze those fields honestly without Phase 43.2 evidence.

Question for Phase 43.2:

- Who owns native output buffers, for how long, and on which side of the host
  ABI?
- What status codes represent cancellation, validation failure, fallback,
  missing toolchain, cache mismatch, and backend divergence?
- Can repeated execution avoid leaks and stale-cache evidence?

### 5. Observability Must Be Contracted Before Freeze

Production native codegen needs diagnostics that name the failing boundary:
verifier, support extraction, MLIR generation, toolchain validation, JIT
execution, output rebuilding, Phase 40 validation, cache admission, route
fallback, or host handoff.

Question for Phase 43.2:

- Are the observability fields stable enough to freeze, or at least to declare
  as versioned ABI fields?

## Proposed 43.2 Acceptance Model

Phase 43.2 should close with a strict gate:

```sh
bash scripts/production-native-codegen-stabilization-test.sh
```

That gate should include:

- deterministic replay and cache-key drift tests;
- production-route native execution tests;
- adversarial output-buffer corruption tests;
- cancellation and missing-toolchain fail-closed tests;
- repeated execution/soak tests over supported Phase 35 rows;
- bounded native-vs-interpreter timing evidence;
- diagnostics/error-code stability checks;
- a guard that skipped native tooling cannot count as positive evidence.

## Phase 44 Handoff

Phase 43.2 should produce a handoff dossier consumed by Phase 44:

- native output buffer shape and ownership;
- Arrow validity/value buffer layout assumptions;
- backend/toolchain/pipeline identity fields;
- runtime/cache key schema;
- validation trace and model/native fingerprint fields;
- fallback and fail-closed status taxonomy;
- cancellation and timeout semantics;
- concurrency/reentrancy assumptions;
- observability/diagnostic fields;
- permanent TCB and non-claims.

Only after that dossier exists should Phase 44 freeze the ABI.
