  <name>Task 1: Add LOOM_SIDECAR_ONLY CMake option and sidecar build path</name>
  <files>contrib/duckdb-ext/CMakeLists.txt</files>
  <action>
Update `contrib/duckdb-ext/CMakeLists.txt` to support two build modes: default full build (links `libloom_ffi.a`) and sidecar-only build (links `libloom_sidecar_ffi.a`).

**Add CMake option** near the top (after `project()`):
```cmake
option(LOOM_SIDECAR_ONLY "Build with only the sidecar FFI path (no container or native lowering deps)" OFF)
```

**Sidecar staticlib path variable:**
```cmake
set(LIBLOOM_SIDECAR_FFI "${WORKSPACE_ROOT}/target/release/libloom_sidecar_ffi.a")
```

**Conditional cargo build command.** After the existing `loom_ffi_build` custom target, add a sidecar build target:
```cmake
if(LOOM_SIDECAR_ONLY)
    set(LIBLOOM_FFI "${LIBLOOM_SIDECAR_FFI}")
    add_custom_command(
        OUTPUT "${LIBLOOM_SIDECAR_FFI}"
        COMMAND cargo build -p loom-sidecar-ffi --release
                --manifest-path "${WORKSPACE_CARGO_TOML}"
        WORKING_DIRECTORY "${WORKSPACE_ROOT}"
        COMMENT "Building libloom_sidecar_ffi.a (lean sidecar-only staticlib)"
    )
    add_custom_target(loom_ffi_build ALL
        DEPENDS "${LIBLOOM_SIDECAR_FFI}"
    )
endif()
```

This approach reuses the existing `LIBLOOM_FFI` variable name in the link step below — when `LOOM_SIDECAR_ONLY=ON`, `LIBLOOM_FFI` points to the sidecar staticlib, so all downstream `target_link_libraries` and include logic continues to work.

**Sidecar header include directory.** When `LOOM_SIDECAR_ONLY=ON`, add the sidecar header path:
```cmake
if(LOOM_SIDECAR_ONLY)
    target_include_directories(loom_loadable_extension PRIVATE
        "${WORKSPACE_ROOT}/crates/loom-sidecar-ffi/include"
    )
endif()
```

**LLVM linking** — in sidecar-only mode, no LLVM/MLIR symbols are needed (sidecar FFI has no native lowering). Wrap the LLVM link logic in `if(NOT LOOM_SIDECAR_ONLY)`:
- The `LLVM_CONFIG_EXECUTABLE` search (lines 25-51) should only FATAL_ERROR when NOT sidecar-only
- `target_link_libraries` for `${LLVM_LIBS}` / `${LLVM_SYSTEM_LIBS}` should be conditional
- `target_link_options` for `${LLVM_LDFLAGS}` should be conditional

**Build and verify:**
```bash
# Sidecar-only build
cmake -S contrib/duckdb-ext -B build/sidecar -DLOOM_SIDECAR_ONLY=ON
cmake --build build/sidecar
# Default full build
cmake -S contrib/duckdb-ext -B build/full
cmake --build build/full
```

<task type="auto">
  <name>Task 2: Add sidecar-aware scan path to DuckDB extension C++ source</name>
  <files>contrib/duckdb-ext/loom_extension.cpp</files>
  <action>
Update `loom_extension.cpp` to support sidecar-mode operation when compiled with `LOOM_SIDECAR_ONLY`. The extension must detect the build mode via a preprocessor define and provide appropriate functionality in each mode.

**Add preprocessor guard.** The CMakeLists.txt should pass `-DLOOM_SIDECAR_ONLY` to the compiler when the option is ON. Add to CMakeLists.txt:
```cmake
if(LOOM_SIDECAR_ONLY)
    target_compile_definitions(loom_loadable_extension PRIVATE LOOM_SIDECAR_ONLY)
endif()
```

**Sidecar-mode header include.** In `loom_extension.cpp`, conditionally include the sidecar header:
```cpp
#ifdef LOOM_SIDECAR_ONLY
#include "loom_sidecar.h"
#else
#include "loom.h"
#endif
```

**Sidecar-mode `loom_scan` behavior.** When `LOOM_SIDECAR_ONLY` is defined:

1. In the bind function (`LoomScanBind`): After parsing the file path argument, call `loom_sidecar_extract(path, &overlay_bytes, &overlay_len)`. If return is 0 (success), the file has a sidecar. Call `loom_sidecar_route(overlay_bytes, overlay_len, nullptr, 0, &decision_json)` to get the routing decision.

2. If the routing decision JSON contains `"decision":"LoomNative"`, the sidecar-gated Loom-native track is applicable — but in sidecar-only mode, Arrow decode is not available. Log a diagnostic that full native decode requires the full build and fall back to host native reader. Set the scan function to use DuckDB's own Parquet reader for the file (via `read_parquet` or similar DuckDB internal path, or return an empty result set with a clear diagnostic).

3. If the routing decision JSON contains `"decision":"HostNativeReader"`, or if `loom_sidecar_extract` returns `NoSidecar(5)`: fall back to DuckDB's own Parquet reader. This is the primary code path for the sidecar-only build — the extension acts as a routing gate that transparently delegates to DuckDB's native reader when Loom-native is not suitable.

4. Always call `loom_sidecar_free_bytes(ptr, len)` for any allocated buffers from extract/route.

**Non-sidecar mode behavior** (`#else`): Existing code path unchanged — `loom_decode`, Arrow C Data Interface, DataChunk population all continue as before.

**Minimal implementation note:** The primary deliverable for sidecar mode is the extraction + routing + graceful fallback path. DuckDB's own Parquet reader can be invoked via DuckDB's internal `ParquetScanFunction` or by returning an empty result set with a clear diagnostic that the sidecar-gated path requires the full build for Arrow output. The exact DuckDB-internal Parquet integration can use `auto &context = *input.context;` and DuckDB's table function registration APIs to chain to the built-in parquet scan — or simply return an error message for this initial integration slice since the value proof is the dependency boundary, not full SQL results. Document which approach was taken in the SUMMARY.

Verification: in full mode, existing DuckDB e2e tests continue to pass. In sidecar mode, `LOAD 'loom'; SELECT * FROM loom_scan('path/to/parquet_with_sidecar');` does not crash and either delegates to DuckDB native reader or returns a clear diagnostic.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CMake build system (LOOM_SIDECAR_ONLY flag) | Build-time switch determines which staticlib is linked; incorrect configuration could silently link wrong library |
| C++ → Rust FFI (loom_extension.cpp → loom-sidecar-ffi) | Sidecar extract/route results cross the C ABI; memory ownership (Vec allocated in Rust, freed via loom_sidecar_free_bytes) must be correctly managed |
| DuckDB host → Parquet file system | File paths from SQL queries reach the Parquet reader through the sidecar extract path |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-51-10 | Tampering | `CMakeLists.txt` sidecar build path | mitigate | Cargo build always runs before CMake link (`add_custom_command` dependency chain); stale `.a` cannot be linked; `LOOM_SIDECAR_ONLY=OFF` by default preserves existing behavior |
| T-51-11 | Denial of Service | `loom_extension.cpp` sidecar extract | mitigate | `loom_sidecar_extract` return code is checked before using output pointers; `NoSidecar(5)` and `DecodeFailed(3)` are handled as fail-closed fallback paths; `loom_sidecar_free_bytes` is always called to prevent memory leaks |
| T-51-12 | Information Disclosure | `loom_extension.cpp` decision JSON parsing | mitigate | `loom_sidecar_route` returns a null-terminated C string; the C++ side parses only the `"decision"` field from the JSON — malformed JSON is treated as `HostNativeReader` fallback |
| T-51-13 | Repudiation | Build mode confusion | mitigate | CMake `option(LOOM_SIDECAR_ONLY OFF)` is explicit and off-by-default; sidecar-mode builds emit a distinct compile definition (`-DLOOM_SIDECAR_ONLY`) that the C++ source checks at compile time; runtime cannot accidentally switch modes |
| T-51-SC | Tampering | npm/pip/cargo installs | mitigate | No new external packages; all tooling (cbindgen, CMake, DuckDB headers) already present and audited. slopcheck: not applicable. |
</threat_model>
