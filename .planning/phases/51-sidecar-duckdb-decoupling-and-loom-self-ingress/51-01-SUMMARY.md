# Phase 51 Plan 01: Lean Sidecar FFI and Parquet-Ingress Decoupling Summary

**Created a zero-container-dependency `loom-sidecar-ffi` staticlib crate with 4 C ABI entry points (extract, verify, route, free_bytes) and removed `loom-core` from `loom-parquet-ingress` production dependencies, enabling the DuckDB sidecar path without pulling in loom-container's 19+ modules.**

- `loom-core` removed from `loom-parquet-ingress` production deps, moved to `[dev-dependencies]` — production code clean with only `loom-ir-core` and `loom-source-ingress`

## Decisions Made

- **Direct loom-ir-core (not loom-core):** Avoids Cargo feature unification — if any workspace member enables `loom-container`, all crates depending on `loom-core` would transitively include it. `loom-sidecar-ffi` depends directly on `loom-ir-core` to stay container-free.
- **parquet dependency in sidecar-ffi:** Needed for `ParquetRecordBatchReaderBuilder` to open Parquet files in `loom_sidecar_extract`. This is a legitimate adapter-layer dependency, not a leakage into the IR model.
- **CString memory management:** `loom_sidecar_verify` and `loom_sidecar_route` return C strings via `CString::into_raw()`. No dedicated free function yet — deferred to Phase 51-02/03.
