<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Locally-built extension ↔ prebuilt DuckDB v1.5.3 CLI | The footer metadata (version/platform/abi) is the load-time trust signal; -unsigned bypasses only the signature, not the metadata check |
| Rust staticlib symbols ↔ DuckDB process symbol namespace | Allocator / symbol clashes live here; flat namespace at dlopen |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-ABI | Spoofing / undefined load behavior | Extension footer vs CLI version/platform | mitigate | POST_BUILD append_metadata.cmake stamps duckdb_version=v1.5.3 (semver, not git hash), host platform string, abi_type=CPP — exactly what GetInvalidMetadataError() checks (RESEARCH Q1, Pitfall 1/2/4); CI asserts the load succeeds |
| T-02-ALLOC | Tampering (heap corruption) | Rust staticlib allocator symbols in the extension dylib | mitigate | System allocator set in Phase 1 (CORE-02); new invariants-script guard runs `nm -g libloom_ffi.a` to confirm no rogue defined malloc/free symbols (PITFALLS P5; ROADMAP criterion 4) |
| T-02-DF2 | Tampering (heap) | Release callback firing on the real DuckDB query teardown path | mitigate | Smoke-test runs a full query to completion; clean process exit confirms the OneShotStream/LoomScanState release fired exactly once (DUCK-03; PITFALLS P1 — verified on the real scan path, not just happy-path unit test) |
| T-02-SC2 | Tampering (supply chain) | DuckDB v1.5.3 CLI download in CI/smoke-test | accept | Official DuckDB GitHub release asset at tag v1.5.3 (RESEARCH §Package Legitimacy Audit, Approved); no npm/pip/cargo installs |
</threat_model>
