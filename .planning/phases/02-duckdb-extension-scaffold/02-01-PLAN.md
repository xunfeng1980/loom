<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DuckDB C++ ↔ Rust (loom_decode) | The C ABI seam; loom_decode return code is the only trust signal that the output pointers are valid |
| Rust-owned Arrow buffers ↔ C++ stream/scan-state ownership | Release-callback ownership crosses here; double-free or leak lives at this seam |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-PANIC | Denial-of-service | LoomInit calling loom_decode | mitigate | loom_decode is catch_unwind-wrapped (Phase 1, DUCK-04); LoomInit checks the i32 rc and throws a DuckDB IOException on nonzero — never proceeds to read output pointers (PITFALLS P5) |
| T-02-DF | Tampering (heap) | OneShotStream + LoomScanState release paths | mitigate | Exactly-once release: get_next zeroes the source release pointer on hand-off; destructor only releases if release pointer non-null; stream release nulls private_data (PITFALLS P1) |
| T-02-LIFE | Information disclosure (dangling read) | ArrowSchema lifetime across bind→scan | mitigate | Schema is owned by the OneShotStream and released only in stream release / scan-state destructor, never by reference into a stack frame (PITFALLS P2) |
| T-02-IDX | Tampering / crash | Arrow buffer index assumptions | mitigate | Wave-0 test (Task 1) pins n_buffers==2, buffers[0]=validity, buffers[1]=values before any C++ buffer logic relies on it; arrow_scan (not hand-indexing) performs the conversion |
| T-02-SC | Tampering (supply chain) | DuckDB v1.5.3 binary downloads | accept | All three artifacts are official DuckDB GitHub release assets at tag v1.5.3 (RESEARCH §Package Legitimacy Audit, all Approved); no npm/pip/cargo installs in this phase |
