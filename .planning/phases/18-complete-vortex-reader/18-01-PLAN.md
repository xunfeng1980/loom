<objective>
Define the Phase 18 complete-reader contract: Loom-owned reader facts, recursive layout/dtype/segment fact types, support classification, stable diagnostics, and dependency-boundary gates before expanding real Vortex conversion support.
</objective>

<context>
Phase 15 introduced `VortexIngressReport` / `VortexFileFacts` for one non-null Int32 real-ingress slice. Phase 18 needs a richer complete-reader boundary that can feed Phase 19 solver-backed verification. This plan creates the contract and Rust vocabulary without yet adding broad conversion support.
</context>

<threat_model>
- Threat: "complete reader" is mistaken for arbitrary Vortex decode support.
  Mitigation: write `18-READER-CONTRACT.md` with explicit accepted/unsupported/rejected semantics and a support matrix boundary.
- Threat: Vortex types leak into `loom-core` or `loom-ffi`.
  Mitigation: keep new reader types inside `loom-vortex-ingress`; strengthen dependency guards.
- Threat: old Phase 15 API consumers break.
  Mitigation: preserve existing `inspect_vortex_buffer`, `inspect_vortex_path`, `emit_supported_lmc1_from_vortex_buffer`, and `VortexIngressReport` behavior while adding richer facts.
- Threat: Phase 19 receives shallow facts.
  Mitigation: require recursive dtype/layout/segment fact types in the contract even if later plans fill them incrementally.
</threat_model>
