<objective>
Wire the Vortex side in `loom-fixtures`: `vortex_reader` inspects an in-memory Vortex `BitPackedArray`/`FoRArray` and emits a loom-core `LayoutNode` + raw packed bytes (D-02), flattening `Validity` to `Option<Vec<bool>>` so loom-core stays Vortex-free; an `oracle` module that decodes the same array via Vortex's own `execute`/canonical path; and the four research-flagged Wave-0 BLOCKING checks plus the bitpack/FOR/nullable roundtrip tests that prove loom-core matches Vortex row-for-row (success criteria 1, 2, 3, 5).

Purpose: This closes the loop — it proves the Plan 01 decode core is correct against the real Vortex layout, and it satisfies INPUT-01/02 (a real in-memory Vortex array is read into the decoder, no .vortex file). The Wave-0 checks come FIRST because they de-risk the riskiest assumptions (BufferHandle byte access A1, FL_ORDER correctness, cross-word 11-bit extract, nullable mapping) before the rest of the wiring is trusted.

Output: `vortex_reader.rs`, `oracle.rs`, `wave0_checks.rs` (BLOCKING), `bitpack_roundtrip.rs`, `for_roundtrip.rs`, and the `loom-core` dev-dependency wiring. `loom_decode`/`loom_scan` are NOT touched (D-03 — DuckDB rewire is Phase 5).
</objective>
