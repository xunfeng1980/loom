**The locked FFI contract (Claude's Discretion in CONTEXT, decided here):**
- `loom_decode` — the single `extern "C"` entry point. Signature:
  `pub unsafe extern "C" fn loom_decode(input_ptr: *const u8, input_len: usize, out_array: *mut arrow::ffi::FFI_ArrowArray, out_schema: *mut arrow::ffi::FFI_ArrowSchema) -> i32`
  - **Error strategy:** integer return code. `0` = success; nonzero = error (the body is wrapped in `catch_unwind`, and any caught panic or `Err` maps to a nonzero code). This is the simplest correct strategy for MVP0; an out-param error string can be added later without breaking the ABI (the return type stays `i32`).
  - **No `loom_free` is exported.** Buffer teardown is owned by the Arrow release callback installed by `to_ffi` (the C++ side calls `array.release(&array)` / `schema.release(&schema)`). This is documented as the ownership contract (ARCHITECTURE.md "Ownership and Release Semantics"). Revisit only if a non-Arrow heap allocation ever crosses the boundary.
- `loom_decode_inner` (or similarly named) — the safe Rust inner fn returning `Result<(), LoomError>`; all real work happens here; `loom_decode` only wraps it in `catch_unwind` and maps the result to an `i32`.
- `LoomError` — a small error enum/type used by the inner fn (mapped to nonzero return codes).

- `crates/loom-ffi/build.rs` — runs `cbindgen` to emit `include/loom.h` (CORE-03)

- `crates/loom-ffi/include/loom.h` — generated header (committed so Phase 2 CMake can `target_include_directories` it)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Rust loom-ffi → C++ DuckDB extension (Phase 2 consumer) | the `extern "C"` surface + Arrow C Data Interface; ownership of buffers transfers here via release callback |
| caller-provided raw pointers → loom_decode | untrusted `*mut`/`*const` pointers; null/validity must be checked before deref |
| package registry → build (cbindgen build-dep) | supply-chain surface for the build-time tool |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | Denial of Service | panic unwinding past `extern "C" loom_decode` → process abort | mitigate | Wrap the entire body in `std::panic::catch_unwind`, map caught panic to a nonzero return code; no `.unwrap()`/`.expect()` on the FFI path (DUCK-04, PITFALLS P3). Verified by `panic_does_not_abort` test + grep gate. |
| T-01-06 | Tampering | Arrow release callback double-free / leak (FFI_ArrowArray) | mitigate | Exactly one `std::ptr::write` per produced struct; never clone the FFI struct; never hold a Rust ref after write; release tested to fire exactly once and leave `release == null` (ARROW-03, PITFALLS P1, RUSTSEC-2022-0012). |
| T-01-07 | Tampering | Schema freed before array consumed (independent lifetimes) | mitigate | `FFI_ArrowSchema` moved via its own `ptr::write`, not by reference; roundtrip test imports+releases schema independently (PITFALLS P2). |
| T-01-08 | Information Disclosure / Tampering | null or out-of-bounds caller pointer dereference | mitigate | Null-check `out_array`/`out_schema`/`input_ptr` and return NullPointer code before any deref; reconstruct slice only after the null check (defensive FFI entry). |
| T-01-09 | Tampering | cbindgen redefining FFI_ArrowArray causing ABI struct mismatch C↔Rust | mitigate | cbindgen.toml excludes/opaques the arrow FFI structs; C++ uses arrow-rs's own header; `loom.h` must NOT contain a `FFI_ArrowArray {` definition (PITFALLS integration gotcha). |
| T-01-SC | Tampering | cargo build-dependency (cbindgen) supply chain | mitigate | `cbindgen` pinned to 0.29.3 (STACK.md, official crate, researcher-vetted; not `[ASSUMED]`/`[SUS]`). Cargo.lock committed. No legitimacy checkpoint required (no flagged packages). |

No `high` open threats remain. The only runtime input this phase accepts is ignored by the stub decode (hardcoded array), so input-parsing attack surface is deferred to Phase 3; the FFI memory-safety invariants above are fully mitigated and unit-tested.
</threat_model>
