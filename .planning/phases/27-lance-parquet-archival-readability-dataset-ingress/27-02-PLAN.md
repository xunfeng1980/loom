<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local Parquet file -> Parquet SDK | Untrusted bytes are parsed by the official Parquet reader. |
| Parquet metadata -> SourceFacts | Metadata becomes review facts but does not authorize artifact bytes. |
| Parquet schema -> support classification | Unsupported shapes must not produce partial Loom output. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-27-02-01 | Tampering | `parquet_source_facts_from_path` | mitigate | Map parse/open failures to rejected reports with no facts; do not trust metadata until the reader opens successfully. |
| T-27-02-02 | Information Disclosure | Parquet diagnostics | mitigate | Diagnostics contain stable code/path/message and no credentials, SDK handles, or object-store state. |
| T-27-02-03 | Tampering | Support classification | mitigate | Nullable, nested, logical, and unsupported schemas are fact-bearing unsupported with no bytes. |
| T-27-02-04 | Elevation of Privilege | Generic public API | mitigate | Parquet SDK types stay inside `loom-parquet-ingress`; static guard from Plan 27-01 covers generic/core/ffi surfaces. |
| T-27-SC | Tampering | Cargo installs | mitigate | Package legitimacy audit in `27-RESEARCH.md` approves `parquet`; no suspicious package is planned. |
</threat_model>
