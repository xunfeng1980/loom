# Phase 27 Archival Readability Report

## Executive Summary

Phase 27 applies the Phase 26 source-ingress contract to two local source families: Parquet files and Lance datasets. The accepted slice is intentionally narrow: local files or datasets containing non-null primitive `Int32`, `Int64`, `Float32`, or `Float64` single-column/raw and simple-table shapes. Accepted outputs are verifier-routed `LMC1` artifacts wrapping `LMP1` or `LMT1`, paired with source facts, oracle evidence, and row-equivalence evidence.

The phase does not claim broad source compatibility. Unsupported valid sources are fact-bearing but byte-free, malformed sources expose no trusted facts, and actual older-version evidence requires real source fixture files or directories plus paired verifier-accepted Loom artifacts. A manifest is provenance only; it is not a substitute for actual older-version source fixture files.

## Supported Slice

Accepted evidence is limited to local source paths and Arrow-compatible, non-null primitive `Int32`, `Int64`, `Float32`, and `Float64` shapes. Single-column sources lower to canonical raw `LMP1` payloads wrapped in `LMC1`; simple multi-column tables lower to `LMT1` payloads wrapped in `LMC1`.

The supported slice includes current-version deterministic fixtures and the checked-in actual older-version fixtures listed below. It excludes remote/object-store IO, credential handling, nullable arrays, strings, nested/list/struct data, logical/extension semantics, source-format writer internals, predicate pushdown, projection pushdown as a source claim, split workers, public SQL/API routes, and new native kernels.

## Unsupported and Rejected Matrix

| Source class | Disposition | Evidence rule |
|---|---|---|
| Valid local source with nullable primitive data | Unsupported | Facts may be present; no Loom bytes, verifier acceptance, or oracle acceptance. |
| Valid local source with strings, nested fields, logical-only fields, dictionary semantics, or extension metadata | Unsupported | Facts may be present; no Loom bytes. |
| Malformed Parquet bytes or non-dataset Lance path | Rejected | Diagnostics only; no trusted facts, no Loom bytes, no oracle evidence. |
| Remote URI, object-store path, credential-bearing input, predicate/split execution request | Out of scope / fail closed | No public route is provided in Phase 27. |
| Manifest-only, record-only, or deterministic-record legacy claim without actual older source files | Failing condition | Must not be accepted as legacy readability evidence. |

## Current-Version Evidence

Current-version Parquet evidence uses `parquet` 58.3.0 through `ParquetRecordBatchReaderBuilder` and adapter-local `ArrowWriter` fixtures in tests. Current-version Lance evidence uses `lance` 7.0.0 through `Dataset::open`, `Dataset::write`, and scanner batch collection. For both source families, accepted paths decode source rows into Arrow batches, canonicalize rows into Loom `LMP1` or `LMT1`, wrap with `LMC1`, call `verify_artifact`, and compare decoded Loom rows against source oracle rows.

## Legacy Readability

Legacy readability is proven by reading each actual older-version source with the current source reader, emitting a verifier-accepted Loom artifact with the current adapter, comparing it to the paired checked-in `.loom` artifact, decoding the paired Loom artifact back through Loom, rewriting the decoded rows to the current source format, and reading the rewritten current source back through the current source reader.

The row fixture for both families is:

`[(1, 10, 1.25, 1.5), (2, 20, -2.5, 2.5), (3, 30, 3.75, 3.5)]`

with schema `id:Int32 non-null`, `score:Int64 non-null`, `ratio32:Float32 non-null`, and `ratio64:Float64 non-null`.

## Verifier Evidence

Accepted reports are constructed only after `loom_core::artifact_verifier::verify_artifact` accepts the emitted `LMC1` bytes. The focused gate also runs `cargo test -p loom-core --test artifact_verifier` to keep the underlying verifier proof active during closeout.

The paired older-version Loom artifacts are sibling files, not source-format embeds:

- Parquet paired Loom artifact: `legacy-v1.loom`, verifier accepted as `LMC1(LMT1)`.
- Lance paired Loom artifact: `legacy-v1.loom`, verifier accepted as `LMC1(LMT1)`.

## Oracle Evidence

Parquet oracle evidence uses `SourceOracleStrategy::ArrowScan` with current `ParquetRecordBatchReaderBuilder` output. Lance oracle evidence uses `SourceOracleStrategy::SourceNativeScan` with current Lance dataset scan output. In both cases, tests check row count, null count, schema shape, and exact primitive values against decoded Loom output.

The oracle path is evidence only. It is not the Loom decode path and does not widen public SQL, CLI, DuckDB, FFI, or native-kernel surfaces.

## Dependency and API Boundary

Lance and Parquet direct dependencies are isolated to workspace pins and their adapter manifests. `loom-core`, `loom-ffi`, `loom-source-ingress`, public headers, DuckDB host code, and CLI routes do not depend on or expose Lance/Parquet SDK types.

The closeout gate scans for route-specific source SQL/API markers, object-store and credential control markers, manifest/footer embedding controls, predicate-pushdown controls, split-worker controls, `ArrowArrayStream` surface creep, and new native-kernel public markers in the guarded public/host surfaces.

## Current-Phase Tradeoffs

Phase 27 uses paired Loom artifacts rather than embedding Loom bytes into Parquet footers or Lance manifests. This keeps the proof format-agnostic and avoids source writer internals, at the cost of less integrated archival packaging.

The adapters canonicalize supported source rows into existing Loom raw/table payloads rather than modeling Parquet or Lance physical encodings. This proves archival readability for a narrow value slice while deferring broader format semantics.

## Non-Goals

- No remote IO, object-store policy, credentials, or cloud provider configuration.
- No Iceberg, MCAP, Zarr, LeRobot, StarRocks, or public dataset catalog binding.
- No public `loom_scan_lance`, `loom_scan_parquet`, `loom_ingest_lance`, or `loom_ingest_parquet` route.
- No Parquet footer embedding or Lance manifest embedding.
- No predicate pushdown, source split workers, source-native parallel execution, or public native kernels.
- No broad nested/logical/string/null compatibility claim.
