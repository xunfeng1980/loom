- But wait — `loom-ffi` and `loom-native-melior` currently import `loom_core::container_codec`, `loom_core::verifier`, etc. from loom-core. If we remove those re-exports, loom-ffi would fail to compile.

**CRITICAL DISCOVERY from dependency analysis:**

`loom-ffi` and `loom-native-melior` import modules that the CONTEXT.md categorized as "legacy" but are actually needed in production:
- `loom-ffi/src/ffi.rs` imports: `container_codec`, `l1_model`, `verifier` 
- `loom-ffi/src/duckdb_runtime.rs` imports: `container_codec`, `artifact_verifier`, `native_arrow_semantic`, `runtime_abi`
- `loom-native-melior/src/backend.rs` imports: `production_native_lowering`, `runtime_abi`
- `loom-native-melior/src/pipeline.rs` imports: `arrow_buffer_lowering`
- `loom-native-melior/src/jit.rs` imports: `arrow_buffer_lowering`, `native_arrow_semantic`

After plan 52-01, these modules have MOVED:
- `l1_model`, `runtime_abi`, `native_arrow_semantic`, `production_native_lowering`, `arrow_buffer_lowering` → in loom-common ✅ (production crates can access via loom-core→loom-common)
- `container_codec`, `verifier`, `artifact_verifier` → in contrib/loom-container ❌

So `loom-ffi` imports `container_codec` and `verifier` — both now in contrib. This means `loom-ffi` MUST depend on `loom-container-legacy` (or we must change loom-ffi's code, which violates zero logic changes).

**Resolution:** `loom-ffi` and `loom-native-melior` will depend on BOTH `loom-core` AND `loom-container-legacy`. Then success criterion 5 ("zero contrib in transitives deps") cannot be met as stated.

**Pragmatic approach:** For loom-core, keep re-exports from BOTH loom-common AND loom-container-legacy:
```rust
pub use loom_common::*;
pub use loom_container_legacy::{container_codec, layout_codec, table_codec, descriptor, verifier, artifact_verifier, verified_lineage};
```
And add `loom-container-legacy` as a non-optional dependency of loom-core. This ensures all downstream crates continue to compile via `use loom_core::*`.

Then in a FOLLOW-UP phase, loom-ffi can be refactored to not use container_codec/verifier directly.
