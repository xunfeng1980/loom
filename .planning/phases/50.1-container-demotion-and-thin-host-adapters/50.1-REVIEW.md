# Phase 50.1: Code Review Report — Container Demotion and Thin Host Adapters

**Reviewed:** 2026-06-11
**Depth:** standard
**Files Reviewed:** 34
**Status:** issues_found

## Summary

Review of the 34 files changed during Phase 50.1 (container demotion of LMC2/LMA1 and ingress adapter degradation to thin host adapters). The refactoring is structurally sound — codec retention, `tcb_status`/`artifact_role` field additions, `#[deprecated]` on `SourceIngressAcceptedArtifact`, and removal of `emit_source_ingress_lmc2_from_*` from all three ingress crates are all correctly executed.

One blocking issue: the `complete-vortex-reader-test.sh` script still expects the CLI `--emit-loom` command to produce LMC2 artifact bytes, but the CLI no longer writes any output file. Two warnings: a stale doc comment in Vortex lib.rs referencing a removed function, and the `arrow_semantic_codec.rs` decoder silently swallowing errors in `decode_arrow_semantic_payload`.

## Critical Issues

### CR-01: `complete-vortex-reader-test.sh` expects LMC2 emission from CLI but CLI no longer writes output

**File:** `scripts/complete-vortex-reader-test.sh:116-127`
**Issue:** The script at lines 117-127 runs `ingest-vortex --emit-loom` and expects it to produce a verifiable LMC2 artifact file:

```bash
cargo run -q --bin loom -- ingest-vortex --emit-loom fixtures/vortex/int32-flat.vortex "${emitted}" >/dev/null
test -s "${emitted}" || fail "CLI emit produced no artifact"
...
grep -q "artifact: LMC2" <<<"${verify_output}" \
    || fail "emitted artifact not identified as LMC2"
```

However, in `crates/loom-cli/src/main.rs` lines 141-163, the `--emit-loom` handler only reads the input, calls `source_facts_from_vortex_buffer`, prints facts to stdout, and **never writes to the output path**. The `_output` variable at line 146 is bound but never used:

```rust
let _output = Path::new(&args[1]);  // line 146 — shadowed, never written to
match source_facts_from_vortex_buffer(&bytes) {
    Ok(facts) => {
        println!("input: {}", input.display());
        println!("status: ingress_ok");
        println!("format: {}", facts.identity.format);
        ...
        Ok(())
    }
    Err(report) => { ... }
}
```

The script will fail because:
1. No output file is created, so `test -s "${emitted}"` fails ("CLI emit produced no artifact").
2. Even if an empty file existed, `verify-artifact` would reject it.

This is **acknowledged** in `50.1-03-SUMMARY.md` ("CLI LMC2 emission path broken ... CLI needs separate updates (out of scope for this plan)") but the script was not updated to reflect this. The script gate is left in a knowingly-broken state.

**Fix:** Either:
- (a) Update the CLI to emit actual LMC2 bytes using the dev-time packaging codec (`encode_arrow_semantic_payload` + `wrap_arrow_semantic_payload`) for backward compat, keeping the emit path functional but documented as dev-time-only.
- (b) Update the script to skip the CLI emit section and note the regression until the CLI is refactored.

Recommended path (a) because the CLI is a developer tool and the `verify-artifact` handoff provides useful diagnostics:

```rust
// In crates/loom-cli/src/main.rs, ingest_vortex "emit-loom" handler:
"--emit-loom" => {
    if args.len() != 2 { return Err(usage()); }
    let input = Path::new(&args[0]);
    let output = Path::new(&args[1]);
    let bytes = fs::read(input).map_err(|err| format!("read {}: {err}", input.display()))?;

    // Dev-time packaging: read Vortex via oracle, encode as LMC2, write to output
    let batches = loom_vortex_ingress::vortex_arrow_oracle_batches_from_buffer(&bytes)
        .map_err(|report| format!("Vortex oracle scan failed: {}", report.status.as_str()))?;
    let payload = ArrowSemanticPayload::from_record_batches(&batches)
        .map_err(|e| format!("arrow semantic payload: {e}"))?;
    let artifact_bytes = encode_arrow_semantic_container_payload(&payload)
        .map_err(|e| format!("LMC2 encode: {e}"))?;
    fs::write(output, &artifact_bytes)
        .map_err(|err| format!("write {}: {err}", output.display()))?;

    println!("input: {}", input.display());
    println!("status: ingress_ok");
    println!("bytes: {}", artifact_bytes.len());
    Ok(())
}
```

### WR-02: `decode_arrow_semantic_payload` silently swallows `from_record_batches` errors

**File:** `crates/loom-core/src/arrow_semantic_codec.rs:320-321`
**Issue:** The decoder uses `.or_else()` to fall back to an empty payload:

```rust
let payload = ArrowSemanticPayload::from_record_batches(&batches)
    .or_else(|_| ArrowSemanticPayload::try_new(schema, Vec::new()))?;
```

If `from_record_batches` fails (e.g., schema mismatch between batches, field/column count mismatch), the error is silently discarded and an empty payload is produced. The subsequent `verify_arrow_semantic_payload` at line 322 will catch the empty payload case, but the original error context is lost. This masks the root cause in diagnostics, making debugging difficult.

**Fix:** Preserve the original error in the fallback or bubble it up:

```rust
let payload = ArrowSemanticPayload::from_record_batches(&batches)
    .or_else(|err| {
        // Attempt fallback with empty batches but persist the original error
        // for diagnostic clarity.
        Err(err)
    })?;
```

Or better, log a diagnostic before falling through, since some IPC streams legitimately produce zero batches and the verifier at line 322 will catch structural issues.

## Info

### IN-01: Test files emit deprecation warnings from `SourceIngressAcceptedArtifact`

**File:** All 13+ ingress test files (e.g., `ingress/loom-vortex-ingress/tests/source_ingress_handoff.rs:15`, `ingress/loom-parquet-ingress/tests/source_ingress_handoff.rs:20`, etc.)
**Issue:** All ingress test files import `SourceIngressAcceptedArtifact` from `loom_source_ingress`, which is now marked `#[deprecated(since = "0.1.0")]`. This produces deprecation warnings during every `cargo test` run. Intentional per Plan 50.1-01 but adds noise to test output.

**Fix:** Add `#[allow(deprecated)]` at the module level of test files, or suppress in the test invocation. This is a cosmetic quality issue, not a bug.

### IN-02: CLI `--emit-loom` handler shadows output path without using it

**File:** `crates/loom-cli/src/main.rs:146`
**Issue:** The variable binding `let _output = Path::new(&args[1]);` is created but never used. The leading underscore suppresses the unused-variable warning, but the variable is semantically misleading — it suggests the path will be used for writing when it isn't. This is a code clarity issue.

**Fix:** Remove the unused binding or implement the output-writing logic (see CR-01 fix suggestion).

### IN-03: Three `bind_content_hash_to_*` stubs discard all parameters

**Files:**
- `ingress/loom-vortex-ingress/src/source_contract.rs:66-71`
- `ingress/loom-parquet-ingress/src/source_contract.rs:85-90`
- `ingress/loom-lance-ingress/src/source_contract.rs:112-118`

**Issue:** All three stubs accept `_ir_hash: &str` and `_host_data_range: (u64, u64)` parameters and return `Ok(())` unconditionally. This is by design for Phase 50 placeholder stubs, but consumers calling these functions may incorrectly assume a binding was recorded. The leading underscores prevent dead-code warnings but silently drop data.

**Fix:** Already documented. Downstream consumers in Phase 50 must implement checks that verify bindings are actually recorded before trusting them. No code change needed now — this is a design note for Phase 50.
