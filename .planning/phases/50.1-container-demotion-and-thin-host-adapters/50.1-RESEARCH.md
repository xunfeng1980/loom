# Phase 50.1: Container Demotion and Thin Host Adapters — Research

**Researched:** 2026-06-11
**Domain:** Rust crate refactoring — container format demotion, ingress adapter degradation
**Confidence:** HIGH

## Summary

Phase 50.1 is the first slice of Repositioning Decision Two (§8 item 4, §9 of `docs/repositioning.md`): demote `LMC2`/`LMA1` from top-level distribution format to optional lineage section + dev-time reference packaging, and degrade the three existing ingress crates into thin host adapters that only mount/extract the sidecar overlay and bind the content-hash to host data — never a second IR, never a second decode path.

The dependency root is Phase 49's independent L2Core IR codec (`l2core_codec.rs` with `L2IR` magic, FNV-1a content-hash). After Phase 49, the L2Core IR is an independent artifact decoupled from all containers. Phase 50.1 uses that independence to demote the old container artifacts and strip decode logic from the ingress crates. **This phase does NOT build the sidecar overlay** (that is Phase 50); it removes the obstacle (old containers as top-level) and prepares the ingress crates to become consumers of the future sidecar contract.

**Primary recommendation:** The demotion is primarily a **role/claim/semantics** change, not a code deletion. The existing `arrow_semantic_codec.rs` module must remain (for dev-time packaging and kloom/verifier tests) but must cease being the default output path from all ingress crates. The three `emit_source_ingress_lmc2_from_*` functions in each ingress crate's `source_contract.rs` must be removed or demoted to dev-test-only status, replaced by thin adapter functions that extract sidecar bytes and bind the hash.

## Architectural Responsibility Map

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| L2Core IR codec + content-hash identity | API/Backend (loom-core) | — | In-TCB; decode IR layer; independent artifact |
| LMC2/LMA1 dev-time reference packaging | API/Backend (loom-core) | — | Out-of-TCB; kept only for kloom/verifier tests |
| Host file mounting (open/inspect) | API/Backend (ingress crates) | — | Each adapter owns its host's SDK boundary |
| Sidecar extraction + hash binding | API/Backend (ingress crates) | — | Thin adapter role — no decode logic |
| Arrow materialization (legacy) | API/Backend (ingress crates) | — | Must be removed from adapters; was always host-native-reader work |
| Artifact verification (LMC2/LMA1) | API/Backend (loom-core) | — | Stays for backward compat; facts updated to out-of-TCB |
| DuckDB SQL surface (loom_scan) | API/Backend (loom-ffi) | API/Backend (loom-core) | Route must still accept LMC2 for backward compat |
| Native execution path | API/Backend (loom-native-melior) | API/Backend (loom-core) | Must be re-anchored to accept sidecar model (Phase 50); LMC2 path stays for compat |

### Pattern 2: Container Demotion — Role Change

**What:** `LMC2`/`LMA1` codec stays in `loom-core` but its role changes from "distribution format" to "dev-time packaging." This is enforced by:
1. Updating module docs in `arrow_semantic_codec.rs` to state out-of-TCB status
2. Updating `artifact_verifier.rs` facts to mark LMC2/LMA1 as "out-of-TCB / dev-time packaging"
3. Removing LMC2 from the default source emission path (ingress crates stop emitting LMC2)
4. Keeping LMC2 decode path for backward-compat test fixtures
5. Keeping `wrap_arrow_semantic_payload` / `unwrap_arrow_semantic_payload` for dev-time tests

**Example (conceptual — artifact verifier facts update):**

### Anti-Patterns to Avoid

- **Deleting LMC2/LMA1 code entirely:** The code is used by kloom/verifier tests and backward-compat fixtures. Delete only the *default emission path*, not the codec itself.
- **Making ingress crates depend on sidecar contract before Phase 50:** Phase 50.1 removes the old emission path; the new sidecar contract is Phase 50. Don't build Phase 50's API in 50.1.
- **Duplicating decode logic in thin adapters:** The current ingress crates already separate fact extraction from Arrow materialization. Keep the boundary clean.
- **Changing the L2Core IR codec:** Phase 49 delivered the independent IR. This phase does not modify it — it uses its independence to demote the old containers.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Content-hash binding to host data | Custom hash scheme for each host | Reuse Phase 49's `l2core_program_hash` (FNV-1a over canonical bytes) | Already implemented, deterministic, architecture-independent |
| Sidecar extraction | Custom binary format parser per host | Defer to Phase 50 sidecar overlay contract | Phase 50.1 removes the old path; Phase 50 defines the new one |
| Arrow materialization | Re-implement Arrow reading per host | Each host's native Arrow reader (vortex-array ArrowSession for Vortex; parquet Arrow reader for Parquet; lance Dataset for Lance) | Already working in existing code; kept for dev-test oracles only |
| LMC2 dev-time packaging | Build new test harness format | Keep existing `wrap_arrow_semantic_payload` / `unwrap_arrow_semantic_payload` | Already roundtrip-proven; no reason to replace |

**Key insight:** This phase primarily **removes code** (the default LMC2 emission path from ingress crates) and **changes documentation/claims** (marking LMC2 as out-of-TCB). The heavy lifting of the new sidecar model is Phase 50.

## Common Pitfalls

### Pitfall 1: Breaking existing release gates

**What goes wrong:** Removing `emit_source_ingress_lmc2_from_*` functions breaks release gate scripts that call them directly.
**Why it happens:** Multiple scripts (`scripts/lmc2-arrow-semantic-container-test.sh`, `scripts/duckdb-lmc2-sql-surface-test.sh`, `scripts/mvp0-verify.sh`, `scripts/source-ingress-contract-test.sh`, `scripts/complete-vortex-reader-test.sh`) reference LMC2 emission or verification.
**How to avoid:** 
  1. Audit all scripts before removing any function
  2. Update scripts to use dev-time packaging path instead of production emission path
  3. Ensure backward-compat fixtures still round-trip through the dev-time path
  4. Run full gate (`scripts/mvp0-verify.sh`) after each wave
**Warning signs:** Gate scripts failing with "function not found" or "marker not found" errors.

### Pitfall 2: Orphaning DuckDB SQL surface

**What goes wrong:** The DuckDB `loom_scan(path)` path defaults to `LMC2(LMA1)` artifacts. If `LMC2` is demoted, DuckDB might lose its primary input format.
**Why it happens:** `loom-ffi/src/duckdb_runtime.rs` resolves artifact kind and routes to decode; removing LMC2 acceptance would break existing SQL queries.
**How to avoid:** Keep LMC2 decode path in `loom-ffi` for backward compat. The DuckDB surface should accept both the new sidecar model (when Phase 50 lands) and LMC2 (for existing fixtures). Phase 50.1 does not remove DuckDB LMC2 support — it only stops *emitting* LMC2 from ingress crates.
**Warning signs:** DuckDB SQL smoke tests failing on `loom_scan` calls.

### Pitfall 3: Breaking `native_arrow_semantic.rs`

**What goes wrong:** The native execution path checks `facts.artifact_kind.as_str() == "LMC2" | "LMA1"` and rejects other artifact kinds.
**Why it happens:** Phase 35 and 43.1/43.2 built the native path around LMC2/LMA1 artifacts.
**How to avoid:** For Phase 50.1, keep LMC2/LMA1 acceptance in `native_arrow_semantic.rs` for backward compat. Add a diagnostic note that this path will be superseded by the sidecar-native track in Phase 50. The native path should NOT be blocked — it should still work with LMC2 dev-time packaging.
**Warning signs:** Native execution tests (`native_arrow_semantic_codegen`, `production_arrow_semantic_jit`) failing.

### Pitfall 4: Incomplete degradation — leaving Arrow materialization in adapters

**What goes wrong:** The `emit_source_ingress_lmc2_from_*` functions are removed, but the underlying Arrow materialization code (e.g., `vortex_arrow_oracle_batches_from_buffer` in Vortex ingress) is moved into a different public function rather than actually removed from the adapter's public API.
**Why it happens:** Large code movement without clear boundary between "thin adapter logic" and "decode logic."
**How to avoid:** 
  1. The only public functions in each degraded adapter should be: fact extraction (`*_source_facts_from_*`), sidecar extraction (stub), and hash binding (stub)
  2. Arrow materialization code should be moved to `tests/` or `#[cfg(test)]` modules OR to dev-only CLI tools
  3. Use `#[doc(hidden)]` on any retained internal helpers to prevent accidental public API exposure
**Warning signs:** `grep` for `RecordBatch` or `ArrowSemanticPayload` in ingress crate `src/` files (non-test).

## Open Questions

1. **What exact public API should each thin adapter expose after degradation?**
   - What we know: Each adapter needs `*_source_facts_from_*` (exists), plus sidecar extraction and hash binding stubs (to be added)
   - What's unclear: Whether the existing `source_ingress_report_from_*` functions should be removed entirely or reimplemented as fact-only (no LMC2 emission). Currently they call `emit_source_ingress_lmc2_from_*` internally.
   - Recommendation: Remove the `source_ingress_report_from_*` functions from the public API of each adapter. Keep only fact extraction + sidecar stubs. The `SourceIngressReport` type stays in `loom-source-ingress` for backward compat.

2. **What happens to the CLI's `loom inspect` command for LMC2 artifacts?**
   - What we know: `loom-cli` currently has a `verify-artifact` command that identifies LMC2 artifacts and reports their status
   - What's unclear: Whether the CLI should continue to accept LMC2 artifacts after demotion, or degrade to "unsupported format" warnings
   - Recommendation: Keep CLI LMC2 acceptance for backward compat with existing fixtures. Add a deprecation notice in CLI output.

3. **How should the test fixtures (`loom-fixtures`) be updated?**
   - What we know: `loom-fixtures` generates LMC2/LMA1 test fixtures used by kloom and verifier tests
   - What's unclear: Whether fixture generation should switch to L2Core IR format instead
   - Recommendation: Keep LMC2/LMA1 fixture generation for dev-time reference packaging tests. Add L2Core IR fixture generation as a parallel path. The Phase 50.1 scope does not include fixture format changes.

4. **What exact changes to release gate scripts are needed?**
   - What we know: At least 5 scripts reference LMC2 directly: `lmc2-arrow-semantic-container-test.sh`, `duckdb-lmc2-sql-surface-test.sh`, `mvp0-verify.sh`, `source-ingress-contract-test.sh`, `complete-vortex-reader-test.sh`
   - What's unclear: How many of these need to be removed vs. updated to reflect the new thin adapter API
   - Recommendation: Audit each script during planning. Scripts that test "LMC2 emission from ingress" should be removed/replaced. Scripts that test "LMC2 decode/verify" should stay (backward compat). Scripts that test "DuckDB SQL over LMC2" should stay (backward compat).
