<objective>
Introduce a minimal Loom layout payload codec and wire `loom_decode(input_ptr,
input_len, ...)` to decode payload bytes instead of always returning the Phase 2
hardcoded Int32 array. This plan prepares the FFI path for DuckDB SQL acceptance
while preserving the existing panic-safety and Arrow C Data Interface ownership
contracts.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| C caller input bytes -> loom-ffi | Input pointer/length may be invalid or contain malformed layout payload bytes. |
| layout_codec -> l1_model | Decoded layout fields can be malformed; codec must check lengths and reject truncation. |
| Rust Arrow output -> C Data Interface | Ownership remains with Arrow release callbacks after `ptr::write`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-09 | Denial of Service | layout payload parser | mitigate | Checked length arithmetic and typed malformed-payload errors before slicing. |
| T-05-10 | Denial of Service | FFI panic boundary | mitigate | Preserve `catch_unwind` wrapper and panic sentinel tests. |
| T-05-11 | Tampering | Arrow FFI ownership | mitigate | Continue exactly one `ptr::write` per Arrow struct and release through Arrow callbacks. |

No high open threats remain if FFI tests pass.
</threat_model>
