<objective>
Add verifier-gated programmatic MLIR module construction through `melior` for the same bounded non-null Int32 copy slice proven by Phase 14 textual lowering, while preserving fail-closed rejection before native artifact creation for unsupported or unverifiable programs.
</objective>

<context>
This plan is blocked on 16-01 because it depends on the optional backend crate and report/diagnostic boundary. The output is a programmatic MLIR artifact, not JIT execution.
</context>

<threat_model>
- Threat: accepted-but-unsupported `L2Core` programs produce MLIR artifacts.
  Mitigation: call `verify_l2_core` and `check_lowering_support` before constructing any `melior` module.
- Threat: verifier facts become detached from the verifier report that produced them.
  Mitigation: builder APIs accept `&FullVerificationReport`, not standalone `VerifiedArtifactFacts`.
- Threat: `melior` alpha API or dialect ownership issues leak into default builds.
  Mitigation: place actual `melior` module construction behind a crate feature named `melior`, and keep default tests green without enabling that feature.
</threat_model>
