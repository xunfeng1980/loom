<objective>
Create the Phase 16 backend boundary without making MLIR/LLVM mandatory for the default workspace build. The deliverable is an optional `loom-native-melior` crate with stable report/diagnostic/toolchain facts, a written backend contract, and guardrails proving `loom-core` and `loom-ffi` remain free of MLIR/LLVM/native-backend dependencies.
</objective>

<context>
Phase 16 promotes the Phase 14 textual MLIR spike into a programmatic backend, but the backend must remain verifier-gated, optional, and fail-closed. This first plan deliberately avoids JIT execution. It establishes the boundary that later plans build on.
</context>

<threat_model>
- Threat: MLIR/LLVM becomes a mandatory dependency of `loom-core`, `loom-ffi`, or the default `cargo test --workspace` path.
  Mitigation: keep `melior`, `mlir-sys`, `llvm-sys`, and native backend code isolated to `crates/loom-native-melior`, with default features that do not require an installed LLVM toolchain.
- Threat: local toolchain mismatch is hidden until execution time.
  Mitigation: add explicit `MlirToolchainFacts` and diagnostics for `llvm-config`, `mlir-opt`, `mlir-translate`, `lli`, detected version, expected version, and compatibility status.
- Threat: backend reports expose unstable third-party types.
  Mitigation: report structs use Loom-owned primitive fields, strings, enums, and vectors only.
</threat_model>
