<objective>
Define the runtime cache identity, stable diagnostics, and a host-neutral C ABI
sketch without implementing a host engine.
</objective>

<context>
Phase 23 needs a backend target and Phase 24 needs a DuckDB adapter target. Both
need a stable cache-key and diagnostic vocabulary before native artifacts are
compiled or reused.
</context>

<threat_model>
- Threat: stale native artifacts are reused across changed verifier facts,
  toolchains, projections, predicates, or CPU policies.
  Mitigation: cache key includes all safety and execution-affecting inputs.
- Threat: diagnostics become host-specific strings.
  Mitigation: use stable diagnostic codes and paths.
- Threat: a C ABI sketch locks in Rust or DuckDB ownership assumptions.
  Mitigation: use opaque handles and plain C-compatible values only.
</threat_model>
