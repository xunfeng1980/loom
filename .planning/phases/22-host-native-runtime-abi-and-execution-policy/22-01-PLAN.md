<objective>
Define the host-neutral Loom runtime lifecycle and model vocabulary before any
host engine or production backend consumes it.
</objective>

<context>
Phase 22 research recommends a Loom-owned runtime contract around verified
artifacts, runtime plans, scan handles, worker-local state, batch output,
diagnostics, cache identity, and fallback decisions. The contract must be
engine-independent in shape even though DuckDB is the first planned consumer.
</context>

<threat_model>
- Threat: DuckDB-specific concepts become the generic runtime ABI.
  Mitigation: define Loom-owned types only; no DuckDB or StarRocks type names.
- Threat: Arrow stream lifetime or thread-safety assumptions leak into the
  contract.
  Mitigation: model scan/worker handles explicitly and keep Arrow export as a
  batch boundary.
- Threat: native execution becomes an unverified side path.
  Mitigation: runtime planning must reference verifier and lowering facts.
</threat_model>
