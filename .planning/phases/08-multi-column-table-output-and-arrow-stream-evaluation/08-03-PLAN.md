# Phase 08-03 Plan: DuckDB Multi-Column Scan and ArrowArrayStream Decision

## Purpose

Extend `loom_scan` from one generic `value` column to named multi-column output, and make the ArrowArrayStream decision with implementation evidence.

## Tasks

1. Update Rust/FFI boundary as needed.
   - Expose enough metadata for DuckDB bind to know column names and types.
   - Preserve single-column FFI decode behavior.
   - Prefer small, explicit metadata functions if table schema cannot fit the existing `loom_decode` signature.
2. Update DuckDB bind/init/scan.
   - Detect single-column vs table payload.
   - Bind named columns for table payloads.
   - Populate `DataChunk` columns for Int32, Boolean, and Utf8.
   - Preserve current single-column `value` behavior.
3. Evaluate ArrowArrayStream.
   - Try the table-shaped output path against the vendored DuckDB headers.
   - If it is stable and simpler, implement it.
   - If it is blocked or fragile, keep direct DataChunk population and document the exact reason.
4. Add DuckDB smoke checks.
   - Load table payload.
   - Select all columns.
   - Run projection/filter/aggregate queries.

## Acceptance Criteria

- `loom_scan(<table-payload>)` returns multiple named columns.
- Single-column `loom_scan(<single-payload>)` still works.
- ArrowArrayStream decision is recorded in summary with evidence.
