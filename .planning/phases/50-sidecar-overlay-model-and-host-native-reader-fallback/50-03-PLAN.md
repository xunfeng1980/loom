<objective>
Implement the fail-closed routing gate (`loom-core/src/sidecar_routing.rs`) that decides: Loom-native track or host-native reader fallback. Add content-hash verification helpers for host data byte ranges to `sidecar.rs`. The routing is exhaustive and honest — every code path returns a specific decision with typed diagnostics.

Purpose: This is the central safety guarantee of the repositioning: an incorrect routing decision either runs unsupported IR through native (EoP risk) or falls back unnecessarily (lost acceleration). The 4-gate logic mirrors the existing `runtime_abi::decide_runtime_execution` pattern for consistency.

Output: `crates/loom-ir-core/src/sidecar_routing.rs` with routing logic + tests; content-hash helpers in `sidecar.rs`.
</objective>

       - `engine_integrated: bool` — whether the calling engine has Loom scan integrated

       - `HostNativeReader { reason: HostNativeReaderReason, diagnostics: Vec<SidecarDiagnostic> }`

    4. **HostNativeReaderReason** — enum:
       - `EngineNotIntegrated`
       - `NoSidecarPresent`
       - `HashMismatch`
       - `EncodingUnsupported`

       - `message: String`

    6. **SidecarDiagnosticCode** — enum mirroring HostNativeReaderReason:
       - `EngineNotIntegrated`
       - `NoSidecarPresent`
       - `HashMismatch`
       - `EncodingUnsupported`

    7. **`pub fn decide_sidecar_routing(input: SidecarRoutingInput) -> SidecarRoutingDecision`**:

       Gate 1: `!input.engine_integrated` → `HostNativeReader { reason: EngineNotIntegrated, diagnostics: [SidecarDiagnostic { code: EngineNotIntegrated, path: "$.engine", message: "calling engine has not integrated Loom scan — fall back to host native reader" }] }`

       Gate 2: `sidecar = match input.sidecar { Some(s) => s, None => return HostNativeReader { reason: NoSidecarPresent, diagnostics: [...] } }`

       Gate 3: Collect all `HashVerificationResult` where `!matches`. If any mismatches exist → return `HostNativeReader { reason: HashMismatch, diagnostics: [one per mismatch] }`.

       Gate 4: `if !input.encoding_supported { return HostNativeReader { reason: EncodingUnsupported, diagnostics: [...] } }`

       All gates pass: collect all bindings from `hash_verification`, return `LoomNative { sidecar, verified_bindings }`.

    1. **`pub fn compute_chunk_hash(data: &[u8]) -> String`**:
       - Compute FNV-1a 64-bit hash over `data` using the same hasher as `l2core_program_hash` (from `l2core_codec.rs`).
       - If the l2core_codec module uses `fnv::FnvHasher` with `std::hash::Hasher`, replicate that pattern:
         ```rust
         let mut hasher = fnv::FnvHasher::default();
         std::hash::Hasher::write(&mut hasher, data);
         let hash = std::hash::Hasher::finish(&hasher);
         ```
         (Check the exact hasher initialization in `l2core_program_hash` — use the same approach.)
       - Format as `"l2ir:<hex>"` where hex is the 64-bit hash in lowercase hex (16 chars, zero-padded).
       - This function hashes raw data bytes — it does NOT encode into L2Core wire format first. It hashes the host file's compressed data pages directly.

    2. **`pub fn verify_chunk_binding(binding: &ChunkBinding, host_data: &[u8]) -> HashVerificationResult`**:
       - Take a `ChunkBinding` and the actual host data bytes for that granule's range.
       - Call `compute_chunk_hash(host_data)`.
       - Compare recomputed hash with `binding.content_hash`.
       - Return `HashVerificationResult { granule_id: binding.granule_id.clone(), binding: binding.clone(), recomputed_hash, matches }`.
       - Note: `HashVerificationResult` is defined in `sidecar_routing.rs`. Either:
         - Move `HashVerificationResult` to `sidecar.rs` (preferred — keeps hash-related types together), and have `sidecar_routing.rs` import it.
         - Or keep it in `sidecar_routing.rs` and add a dependency edge. Choose based on cleaner module boundaries: `HashVerificationResult` belongs with hash computation, so define it in `sidecar.rs` and import in `sidecar_routing.rs`.

    3. **Update the `sidecar_routing.rs` import** to use `HashVerificationResult` from `sidecar`.

    Do NOT:
    - Add a new hashing algorithm — use FNV-1a exclusively to match the L2Core IR content-hash
    - Hash the sidecar metadata itself — hash only the host data ranges
    - Implement cryptographic hashing (SHA-256 deferred per RESEARCH.md)
    - Change `l2core_program_hash` — that is Phase 49 territory

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| decide_sidecar_routing input | Untrusted: engine_integrated, sidecar bytes from host metadata, hash verification results from external computation |
| LoomNative → native execution | Trust elevation: only LoomNative decisions proceed to Loom verifiable-native track; HostNativeReader decisions bypass Loom entirely |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-50-06 | Elevation of Privilege | sidecar_routing.rs gate bypass | mitigate | Exhaustive match: every code path returns a decision; no fallthrough; Gate 4 (encoding_supported) prevents unsupported IR from reaching native execution |
| T-50-07 | Spoofing | sidecar_routing.rs HashMismatch gate | mitigate | Gate 3 requires ALL bindings to match; a single mismatch fails the entire routing to HostNativeReader; no partial-LoomNative path |
| T-50-08 | Tampering | sidecar.rs compute_chunk_hash | accept | FNV-1a is non-cryptographic and trivially collisionable; threat model is non-adversarial (sidecar authored by same party as host data) per RESEARCH.md A2; collision risk documented; SHA-256 deferred |
| T-50-09 | Denial of Service | sidecar.rs compute_chunk_hash over large host data | mitigate | Hash computation is O(n) over host data bytes; column chunks typically ≤ hundreds of MB; FNV-1a is fast; hash cost is documented as sub-dominant relative to native decode benefit; RESEARCH.md Pitfall 2 suggests configurable fast-mode hashing — implement hashing over full range but add comment noting future optimization path |
| T-50-10 | Information Disclosure | sidecar_routing.rs diagnostics | mitigate | Diagnostics contain only granule_id, expected/recomputed hash (non-sensitive), and routing reason codes; no host file path, IR content, or engine internals exposed |
</threat_model>
