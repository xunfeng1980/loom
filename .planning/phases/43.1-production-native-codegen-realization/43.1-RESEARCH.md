# Phase 43.1 Research: Production Native Codegen Realization

## Findings

### 1. The Phase 35 output path is real execution, but not real codegen.

`execute_verified_native_arrow_semantic` verifies `LMC2` / `LMA1`, decodes the
Arrow semantic payload, and builds a new RecordBatch through typed Arrow builder
copy helpers. That is valid Phase 35 evidence, but it bypasses the
MLIR/LLVM/JIT/native backend and therefore cannot satisfy CODEGEN-01.

### 2. Phase 40 can already validate an externally produced native output.

`verify_native_arrow_semantic_model_output` compares a supplied RecordBatch with
the Phase 39 reference model trace and decoded Arrow value equality. Phase 43.1
should reuse that as the admission gate for JIT-produced output rather than
inventing a second validation vocabulary.

### 3. The production JIT seed is useful but too narrow.

`execute_prepared_production_jit` already checks backend report acceptance,
entry symbol, toolchain compatibility, cancellation, explicit input buffers,
MLIR lowering, and ExecutionEngine invocation. The current raw-copy JIT can
return value buffers for Int32, Int64, Float32, and Float64.

Missing for Phase 43.1:

- Boolean value buffer support.
- Nullable validity bitmap input/output support.
- Arrow semantic payload support in lowering facts.
- RecordBatch assembly from JIT-produced buffers.
- Validation-gated runtime/cache admission with production backend identity.

### 4. Existing production lowering facts reject the needed payload.

`check_production_lowering_support` expects `LMP1 layout` or `LMT1 table` plus
L2 facts. Phase 43.1 needs a separate Arrow semantic codegen support path that
starts from verifier-accepted `Arrow semantic payload` facts and decoded Arrow
schema/value buffers.

### 5. The buffer model needs to stop being value-only.

`ArrowColumnBufferPlan` already has a validity plan type, but `plan_column`
rejects nullable columns. The JIT API only accepts value buffers. Phase 43.1
needs a native column input/output model that carries:

- value buffer bytes;
- optional validity bitmap bytes;
- null count;
- Arrow type;
- row count;
- field name and schema metadata needed to rebuild a RecordBatch.

### 6. Boolean is a special case.

Arrow Boolean values are bitmap-backed, not byte-per-row primitives. For Phase
43.1, Boolean support should be modeled as byte-addressed bitmap copy in MLIR
over the exact Arrow value buffer byte length, with validity bitmap handled the
same way. Positive tests must include non-zero, non-trivial bit patterns so an
all-zero placeholder cannot pass.

## Recommended Architecture

1. Add an Arrow semantic native-codegen support adapter in `loom-core`.
   It should verify the artifact, decode the one-batch Arrow semantic payload,
   classify the Phase 35 matrix, and extract deterministic column buffer inputs.

2. Extend the Arrow buffer lowering plan to represent nullable value+validity
   buffers and Boolean bitmap buffers.

3. Extend `loom-native-melior` production JIT to copy both typed primitive value
   buffers and byte-addressed bitmap buffers through MLIR/LLVM/JIT.

4. Add a production Arrow semantic codegen execution report that assembles the
   JIT output into a RecordBatch and then calls Phase 40 model validation.

5. Only validated output may become a native runtime candidate or cache key.
   Divergence, unsupported shapes, missing/failed toolchain, and skipped
   toolchain must fail closed or take explicit interpreter fallback according
   to runtime policy.

6. Record the resulting backend/toolchain/pipeline/cache identity as Phase 44
   ABI freeze input.

## Risks

- MLIR toolchain availability can make local positive JIT tests environment
  sensitive. The phase should keep broad gates skip-aware, but the focused
  Phase 43.1 positive gate must not count toolchain skip as success.
- Boolean and validity bitmaps are byte-buffer operations. Off-by-one errors at
  non-multiple-of-eight row counts are the highest-risk correctness bug.
- Cache identity must include toolchain and trace fingerprints, or Phase 44 may
  freeze an ABI that cannot distinguish semantically different native outputs.
- The old builder-copy API should remain available as interpreter/fallback
  evidence, but its backend naming must not obscure production-codegen evidence.
