## Decision IDs Used In Phase 29 Plans

- D-01: Keep DuckDB executable evidence on existing public `loom_scan(path)`.
- D-02: Add only an adapter-local offline StarRocks-compatible descriptor/query contract by default.
- D-03: Use Phase 28 accepted local Iceberg binding plus verifier-accepted Loom artifact as the shared input.
- D-04: Keep the initial query matrix to projection, filter-like predicate, count, sum, and deterministic row materialization.
- D-05: Sidecar claims alone are not sufficient for either surface.
- D-06: StarRocks-compatible descriptors are accepted only when identity fields match the accepted binding.
- D-07: DuckDB evidence must be executable; StarRocks evidence may be local descriptor validation.
- D-08: Fail closed on stale binding evidence, schema/snapshot/hash mismatch, unsupported routes, and unsupported query features.
- D-09: Do not add StarRocks server/client/JDBC/ODBC/REST/credential/catalog/object-store dependencies by default.
- D-10: Keep core, FFI, source-ingress, and Phase 28 binding semantics engine-neutral.
- D-11: Do not change public C ABI or DuckDB public SQL.
- D-12: Do not broaden Vortex semantics, distributed execution, predicate pushdown, or Iceberg catalog commit scope.
- D-13: Produce `29-DUAL-QUERY-SURFACE-REPORT.md`.
- D-14: Add `scripts/dual-query-surface-test.sh` after Phase 28 and before DuckDB smoke.
- D-15: Include negative tests for scope creep, separate artifact formats, unchecked sidecar success, stale identity, and unsupported query features.
- D-16: Keep the release proof deterministic on a clean checkout.
- D-17/D-18/D-19/D-20: Record current-phase tradeoffs: offline StarRocks contract, existing DuckDB surface, small query matrix, adapter-local descriptors.
- D-21: The agent may choose exact crate/module names and descriptor JSON shape; this plan chooses `loom-dual-query-surface`.
