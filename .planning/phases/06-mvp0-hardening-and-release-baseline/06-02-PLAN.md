# Phase 06-02 Plan: One-Command MVP0 Verification Gate

## Purpose

Add a repository-root release gate that runs the full MVP0 acceptance suite without requiring the user to remember separate commands or clean build artifacts manually.

## Tasks

1. Add `scripts/mvp0-verify.sh`.
   - Use `set -euo pipefail`.
   - Resolve `REPO_ROOT` via `git rev-parse --show-toplevel`.
   - Run from the repository root even if invoked from another directory.
   - Print concise step labels.

2. Gate contents:
   - `cargo test --workspace`
   - `cargo tree -p loom-core | awk '/vortex|fastlanes/{c++} END{print c+0}'` and fail unless output is `0`
   - `rg -n 'vortex_file|vortex-file|\\.vortex|VortexFile|from_path|read_file' crates/loom-fixtures` and fail if matches exist
   - `bash scripts/duckdb-smoke-test.sh`

3. Strengthen stale-artifact behavior.
   - Keep `scripts/duckdb-smoke-test.sh`'s explicit `cargo build -p loom-ffi --release`.
   - Ensure old `duckdb-ext/build/loom.duckdb_extension` is removed before relink.
   - If needed, update `duckdb-ext/CMakeLists.txt` so a direct `cmake --build duckdb-ext/build` has a clearer dependency on the Rust staticlib build step.
   - Avoid over-engineering CMake; the release gate is the primary acceptance path.

4. Add shell-script hygiene.
   - Make `scripts/mvp0-verify.sh` executable.
   - Keep output stable enough to paste into summaries.
   - Do not require network if the DuckDB CLI is already cached or `DUCKDB_CLI` is set; preserve current smoke-test behavior.

## Acceptance Criteria

- `bash scripts/mvp0-verify.sh` passes from repo root.
- `cd crates/loom-core && bash ../../scripts/mvp0-verify.sh` also runs against the repo root.
- A nonzero `loom-core` Vortex/FastLanes dependency count fails the script.
- A forbidden file-backed Vortex API match in `crates/loom-fixtures` would fail the script.

## Verification

- `bash scripts/mvp0-verify.sh`
- `cd crates/loom-core && bash ../../scripts/mvp0-verify.sh`
- `git diff --check`
