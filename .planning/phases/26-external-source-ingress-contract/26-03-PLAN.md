<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| source bytes -> Loom artifact bytes | Untrusted source bytes may produce `.loom` bytes only through supported conversion and verifier acceptance. |
| emitted LMC1 -> SourceIngressReport accepted | Artifact verification facts become the acceptance gate for source ingress. |
| source-native oracle -> evidence metadata | Oracle output is evidence for tests, not the implementation path or a trust bypass. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-26-03-01 | Tampering | accepted source artifact | mitigate | Require `verify_artifact` accepted status before `SourceIngressStatus::Accepted`. |
| T-26-03-02 | Spoofing | oracle evidence | mitigate | Record oracle strategy/status separately and keep Vortex scan as evidence only; tests decode verifier-accepted Loom bytes. |
| T-26-03-03 | Elevation of Privilege | unsupported/rejected sources | mitigate | Unsupported valid emits no bytes; rejected carries no facts; tests cover both. |
| T-26-SC | Tampering | package installs/dependencies | mitigate | No package installs are planned; use existing Vortex and loom-core dependencies only inside the source-specific crate. |
</threat_model>
